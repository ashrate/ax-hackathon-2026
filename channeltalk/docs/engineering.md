# 채널톡 — 엔지니어링 기록

> 이 파일은 submission.zip에는 포함되지 않는 작업 기록이며, 문항 ③·④와 README.md 작성의 원천 자료다.

## 1. 플러그인 설계 개요

- 플러그인명: ChannelTalk ALF Knowledge Planner
- 해결 문제: 이커머스 운영자와 CX/CS 담당자가 공개 FAQ, 배송·교환·환불 정책, 상품 안내, 쿠폰/프로모션 정보를 ALF용 지식·규칙·테스트로 바꾸는 작업이 수작업이다.
- 스킬 구성: `src/skills/main/SKILL.md` 하나로 실행한다. 근거 맵은 `src/references/evidence.md`에 둔다.
- 핵심 입력: 공개 URL, 붙여넣은 FAQ/정책/상품 설명, 대상 범위(배송, 교환/환불, 주문 변경, 사이즈, 쿠폰, 상담원 이관 등).
- 핵심 출력: `output/<브랜드명>/alf-knowledge/`, `alf-rules.md`, `alf-test-cases.csv`, `alf-escalation.md`, `gaps.md`, `APPLY-GUIDE.md`.

## 2. 출력 계약

2026-07-08 새 최우선 지침에 따라 “감사 문서가 아니라 결과물”로 범위를 고정했다. 사용자가 문제를 던졌을 때 플러그인은 보고서가 아니라 `output/<브랜드명>/` 폴더를 만든다.

계약 파일:

- `docs/output-spec.md`: 산출물별 스키마, 품질 기준, 정보 부족 시 동작.
- `src/skills/main/SKILL.md`: 입력 수집 → 공개 문서 수집·분류 → 계약된 출력 생성 → 자체 품질 점검.
- `examples/biteme/`: 채널톡 공식 사례에 등장한 바잇미 공개 자료로 생성한 실제 산출물.

## 3. 작동 절차

1. 사용자가 준 브랜드명과 URL 목록을 입력 계약으로 정리한다.
2. 공개 URL을 FAQ, 배송, 교환/반품/환불, 주문/회원, 쿠폰/적립금, 상품별 정책으로 분류한다.
3. 로그인 없이 읽을 수 있는 본문과 공개 AJAX 응답만 사용한다.
4. `alf-knowledge/` 지식 문서를 카테고리별로 만든다.
5. `alf-rules.md`, `alf-escalation.md`, `alf-test-cases.csv`, `gaps.md`, `APPLY-GUIDE.md`를 생성한다.
6. 지식 문서의 출처 URL/확인일 표기, CSV 파싱 가능 여부, 테스트 질문 수, 고위험 주제 이관 처리를 자동 스캔과 수동 검수로 확인한다.

## 4. 판단 기준

- 공식 영상 자막의 “이커머스 고객사의 상담 문제”와 “AI 에이전트” 힌트를 최우선으로 둔다.
- 회사 공식 자료와 제품 문서는 보조 자료보다 우선한다.
- 고객에게 손해나 오해를 줄 수 있는 정책 항목은 사람이 최종 확인해야 한다.
- 출처가 불명확하거나 상충하면 답을 생성하지 않고 확인 질문을 만든다.
- 내부 ALF 설정, 고객사 주문 데이터, CRM 데이터 접근을 전제하지 않는다.

## 5. 정보 부족·실패 상황 폴백

- URL 접근 실패: 사용자가 텍스트를 붙여넣도록 요청하고, `gaps.md`의 접근 실패 URL에 기록한다.
- 문서 부족: `gaps.md`의 미확인 정책과 운영자 확인 질문으로 출력한다.
- 문서 충돌: 충돌하는 문장과 URL을 나란히 보여주고 사람 검토로 넘긴다.
- 정책 민감 질문: 답변 초안 대신 확인 질문과 이관 기준을 먼저 낸다.
- 공개 근거 부족: 제출 답변에는 포함하지 않고 “보조 가설”로만 둔다.

## 7. AI 활용 기록

- AI에 맡긴 작업: 자막·공개 자료 요약, 후보 문제 비교, 스캐폴딩 구조 제안, README/SKILL/검증 문서 정합성 점검.
- 직접 판단한 부분: 채널톡 트랙은 설치 QA보다 이커머스 상담 AI 준비 문제를 중심으로 잡는 것이 영상 의도와 더 맞다고 결정했다.
- 받아들이지 않은 AI 제안: 공개 리뷰 VOC 티켓화는 구현 가능하지만 채널톡 고객사의 상담 문제보다 주변 문제라 최종안에서 제외했다.
- 이번 근거 감사에서 AI에 맡긴 작업: 공식 도움말/뉴스룸/개발자 문서 검색, URL 접근성 확인, 핵심 주장별 등급 표 작성, 반증 후보 정리.
- 이번 근거 감사에서 직접 판단한 부분: 보조 기사와 리뷰를 문항 ② 핵심 근거에서 내리고, 공식 제품/도움말/뉴스룸을 최종 출처 후보로 확정했다.
- 이번 근거 감사에서 받아들이지 않은 제안: "채널톡이 이미 AI 상담 시뮬레이션을 제공하므로 문제를 다시 고르자"는 방향은 채택하지 않았다. 이번 라운드는 재선정이 아니라 보강이며, 플러그인의 역할을 제품 기능 대체가 아닌 공개 자료 기반 사전 설계로 좁히면 기존 문제 정의가 유지된다고 판단했다.
- 이번 출력 계약 전환에서 AI에 맡긴 작업: 출력물 스키마 작성, 바잇미 공개 FAQ/상품 상세 수집, 예시 산출물 초안 작성, CSV 파싱 검증.
- 이번 출력 계약 전환에서 직접 판단한 부분: 결과물을 `alf-knowledge/`, 규칙, CSV 테스트, 이관 기준, gaps, 적용 가이드로 고정하고 API 키 없는 공개 문서 기반 프로토타입으로 제한했다.
- 이번 출력 계약 전환에서 받아들이지 않은 제안: 실제 채널톡 관리자 화면 자동 반영이나 주문/환불 처리 자동화는 API 키와 내부 데이터가 필요하므로 제외했다.

## 8. 외부 리뷰 피드백 수용과 반영

- 받은 피드백: 채널톡 트랙은 3트랙 중 제출 가능성이 가장 높지만, 바잇미 한 곳만으로는 이커머스 일반화 증명이 약하고 README 전면부가 의사결정 감소 효과를 빠르게 보여주지 못한다.
- AI에 맡긴 작업: 바잇미와 다른 업종의 공개 FAQ 접근 가능 후보를 탐색하고, 알라딘 고객센터 FAQ/AJAX 응답 URL을 `curl -L`로 점검하고, 라이트 실행 산출물 초안을 만들게 했다.
- 직접 판단한 부분: 알라딘을 브랜드2로 확정했다. 바잇미는 반려동물 커머스이고 알라딘은 도서/음반/전자책 커머스라 업종이 다르며, FAQ 목록과 배송/반품/교환 AJAX 응답이 로그인 없이 접근 가능했기 때문이다.
- 반영한 수정: `examples/aladin/`에 지식 맵 요약, 대표 지식 문서 3개, 규칙 발췌, 12개 테스트 CSV, gaps를 추가했다. README 전면부는 입력 예시, 실행 결과, 실패 시 동작, 사람이 최종 판단하는 지점 순서로 재배치했다.
- 표현 수정: 무근거 주장이 없다고 단정하는 문구는 검사 방법이 붙은 표현으로 바꿨다. 예시는 "지식 문서의 출처 URL/확인일 표기, CSV 파싱 가능 여부, 테스트 질문 수, 고위험 주제 이관 처리를 자동 스캔과 수동 검수로 확인"이다.
- 받아들이지 않은 제안: 알라딘 라이트 실행을 바잇미와 같은 30개 이상 테스트의 풀 실행으로 확장하는 방향은 이번 보완에서 제외했다. 리뷰 요구는 일반화 증명용 라이트 실행이었고, 제출 전 남은 시간에서는 README 전면부와 출처 점검을 함께 끝내는 것이 더 중요하다고 판단했다.
- 검증: README, docs, src, examples의 인용 URL 53개를 중복 제거 후 `curl -L --max-time 20`로 점검했고 HTTP 200 응답을 확인했다. 플러그인 validator는 별도 제출 전 검증 단계에서 다시 실행한다.

## 9. 답변 초안

### 문항 ③ 플러그인은 어떻게 작동하나요? (1000자 제한)

이 플러그인은 사용자가 넣은 이커머스 FAQ, 배송·교환·환불 정책, 상품 설명, 쿠폰 안내 같은 공개 문서를 ALF 도입 전 준비물로 바꿉니다. 먼저 URL과 텍스트를 출처 단위로 기록하고, 문서를 배송, 교환/환불, 주문 변경, 상품/사이즈, 쿠폰, 상담원 이관 같은 상담 의도별로 나눕니다. 각 의도마다 답변에 필요한 근거 문장, AI가 말해도 되는 규칙, 금지 답변, 상담원 이관 기준을 만듭니다. 환불, 취소, 배송 예정일처럼 고객 오해가 큰 항목은 high risk로 표시하고 사람이 확인해야 할 질문을 붙입니다. 문서가 부족하면 추정하지 않고 누락 체크리스트를 만들며, 문서끼리 충돌하면 conflict로 표시하고 사람 검토로 넘깁니다. 마지막으로 정상 질문, 애매한 질문, 정책 충돌 질문, 이관 질문을 포함한 ALF 테스트 세트를 출력합니다.

### 문항 ④ AI를 어떻게 활용했나요? (800자 제한)

AI에는 유튜브 자막과 공개 자료 요약, 후보 문제 비교, 출처별 근거 표 정리, 플러그인 작동 절차 초안, README와 SKILL.md의 정합성 점검을 맡겼습니다. 직접 판단한 부분은 최종 문제 선택입니다. 채널톡 후보로 설치 QA, 공개 리뷰 VOC 정리, ALF 준비가 있었지만, 출제 영상이 이커머스 상담 문제와 AI 에이전트를 직접 언급했기 때문에 ALF 지식·규칙·테스트 구조화 문제를 선택했습니다. 받아들이지 않은 제안은 리뷰 VOC 티켓화입니다. 공개 리뷰 기반이라 검증은 쉽지만 채널톡 고객사의 상담 자동화 문제보다 주변 문제라 대회 의도와 약하다고 봤습니다. 막힌 지점은 공개 자료만으로 내부 ALF 설정을 알 수 없다는 점이었고, 이를 내부 설정 자동화가 아니라 공개 문서를 바탕으로 한 준비 체크리스트와 테스트 세트 생성으로 범위를 줄여 해결했습니다.
