# 채널톡 플러그인 출력 계약

확정일: 2026-07-08

## 목적

이 플러그인은 감사 문서가 아니라 채널톡 ALF 설정 준비물을 만든다. 사용자가 이커머스 브랜드 홈페이지, FAQ, 배송/교환/환불 정책 URL 목록을 주면 Codex는 공개 문서를 수집하고 `output/<브랜드명>/` 아래에 ALF 지식, 규칙, 테스트 케이스, 이관 기준, 누락 질문, 적용 가이드를 생성한다.

예시 실행은 `examples/<브랜드명>/`에도 같은 구조로 남긴다. `<브랜드명>`은 파일 시스템에서 안전한 영문 소문자 슬러그를 권장하되, 문서 제목에는 실제 브랜드명을 쓴다.

## 디렉터리 계약

```text
output/<브랜드명>/
  alf-knowledge/
    <지식 카테고리>/
      <문서명>.md
  alf-rules.md
  alf-test-cases.csv
  alf-escalation.md
  gaps.md
  APPLY-GUIDE.md
```

## 입력 계약

필수 입력:

- 브랜드명
- 공개 접근 가능한 URL 1개 이상

권장 입력:

- FAQ URL
- 배송/배송비/배송지연 정책 URL
- 교환/반품/환불 정책 URL
- 주문 취소/주문 변경/회원/쿠폰/적립금 정책 URL
- 상품별로 정책이 달라지는 경우 대표 상품 URL

입력 제한:

- 로그인, 주문번호, 고객 개인정보, API 키, 관리자 화면 캡처는 요구하지 않는다.
- 공개 URL이 막히면 사용자에게 텍스트 붙여넣기를 요청할 수 있지만, 산출물에는 붙여넣은 출처와 확인일을 분리해 남긴다.

## 공통 품질 기준

- 출처 없는 정책 주장 0건.
- 모든 확정 정책 문장에는 출처 URL과 확인일이 있어야 한다.
- 공개 문서가 애매하면 확정 답변 대신 `미확인` 또는 `운영자 확인 필요`로 표시한다.
- URL 접근 실패, 동적 로딩 실패, 본문 일부 미수집은 `gaps.md`에 남긴다.
- 환불, 취소, 결제, 개인정보, 법적 책임, 의료/건강 효능, 상품 하자 판정은 고위험 주제로 보고 보수적으로 답한다.
- 고위험 주제에서 근거가 부족하면 ALF가 단정하지 않고 상담원 이관 또는 추가 확인 질문으로 처리한다.
- 브랜드 내부 주문/CRM/상담 데이터가 필요한 내용은 공개 문서만으로 확정하지 않는다.

## `alf-knowledge/` 스키마

목적: 채널톡 ALF 지식 관리 화면에 폴더째 옮길 수 있는 구조화 문서 묶음.

폴더 규칙:

- 폴더명은 `01-brand-and-scope`, `02-order-and-account`, `03-shipping`, `04-exchange-return-refund`, `05-coupons-points`, `06-contact-and-escalation`처럼 정렬 가능한 번호와 카테고리명을 쓴다.
- 없는 카테고리는 만들지 않아도 되지만, 배송/교환/환불은 가능하면 분리한다.

문서 규칙:

- 파일명은 소문자 케밥 케이스 `.md`를 쓴다.
- 각 문서는 다음 섹션을 가진다.
  - `# 제목`
  - `ALF 답변 범위`
  - `확정 지식`
  - `답변 시 주의`
  - `상담원 확인 필요`
  - `출처`
- `출처` 섹션에는 `출처 URL`, `확인일`, 필요하면 `수집 방법`을 쓴다.
- 정책 문서의 핵심 내용은 ALF가 고객에게 답할 수 있는 문장 단위로 쓴다.
- 서로 다른 상품의 정책이 다르면 공통 정책처럼 합치지 말고 상품별 조건으로 분리한다.

품질 기준:

- 각 문서는 한 카테고리의 고객 질문을 처리할 수 있어야 한다.
- 같은 문서 안에서 일반 규칙과 상품별 예외가 충돌하지 않아야 한다.
- 문서 하단에 출처 URL과 확인일이 없는 문서는 실패로 본다.

## `alf-rules.md` 스키마

목적: 채널톡 규칙 입력창에 복사해 넣을 수 있는 상황별 지시문 블록.

필수 섹션:

- `# ALF Rules - <브랜드명>`
- `기본 톤`
- `공통 금지 답변`
- `정책별 규칙`
- `고위험 주제 처리`
- `출처`

규칙 블록 형식:

```md
## [규칙명]
- 적용 상황:
- ALF 지시문:
- 답변 금지:
- 상담원 이관:
- 근거 URL:
```

품질 기준:

- 규칙은 고객에게 보이는 문장보다 ALF에게 지시하는 문장으로 쓴다.
- “대략”, “아마”, “보통”처럼 근거 없는 완충어로 정책을 단정하지 않는다.
- 상품별 조건이 다른 배송비, 반품 가능 기간, 택배사는 공통 규칙으로 합치지 않는다.

## `alf-test-cases.csv` 스키마

목적: ALF 배포 전 테스트 질문 세트.

CSV 헤더는 정확히 다음 5개 컬럼을 사용한다.

```csv
질문,의도분류,기대답변요지,근거URL,오답위험도
```

컬럼 정의:

- `질문`: 고객이 실제 채팅에 입력할 법한 문장.
- `의도분류`: 배송, 주문취소, 교환, 반품, 환불, 쿠폰, 회원, 상담원 이관 등.
- `기대답변요지`: ALF가 답해야 할 핵심 요지. 개인 주문 정보가 필요하면 확인 요청 또는 이관이라고 쓴다.
- `근거URL`: 공개 URL. 여러 URL이면 공백 없이 `|`로 연결한다.
- `오답위험도`: `낮음`, `중간`, `높음` 중 하나.

품질 기준:

- 최소 30개 이상.
- CSV 파서로 읽을 수 있어야 한다.
- 정상 질문, 애매한 질문, 예외 조건, 상담원 이관 질문을 모두 포함한다.
- 고위험 질문에는 확정 답변보다 확인 조건과 이관 기준이 들어가야 한다.

## `alf-escalation.md` 스키마

목적: 상담원 이관 기준.

필수 섹션:

- `# ALF Escalation - <브랜드명>`
- `즉시 이관 트리거`
- `추가 질문 후 이관`
- `이관 제외 조건`
- `이관 문구`
- `출처`

품질 기준:

- 트리거 조건은 고객 표현, 필요한 정보, 제외 조건을 함께 쓴다.
- ALF가 주문번호, 개인정보, 내부 처리 상태를 추정하지 않도록 한다.
- 이관 문구는 고객에게 책임을 전가하지 않는 문장으로 쓴다.

## `gaps.md` 스키마

목적: 사이트에 없거나 모호해서 운영자가 확인해야 할 질문 목록.

필수 섹션:

- `# Gaps - <브랜드명>`
- `접근 실패 URL`
- `미확인 정책`
- `문서 충돌`
- `운영자 확인 질문`

행 형식:

```md
| 항목 | 상태 | 왜 필요한가 | 확인 요청 문장 | 관련 URL |
```

상태 값:

- `접근 실패`
- `부분 수집`
- `미확인`
- `충돌`

품질 기준:

- 정보 부족을 ALF 규칙으로 숨기지 않는다.
- `gaps.md`의 미확인 항목은 `alf-rules.md`에서 단정 답변 금지 또는 이관 조건으로 연결되어야 한다.

## `APPLY-GUIDE.md` 스키마

목적: 산출물을 채널톡 관리자 화면에 적용하는 단계별 가이드.

필수 섹션:

- `# Apply Guide - <브랜드명>`
- `사전 준비`
- `1. 지식 문서 등록`
- `2. 규칙 등록`
- `3. 상담원 이관 기준 등록`
- `4. 테스트 케이스 실행`
- `5. 운영 전 확인`
- `6. 업데이트 주기`

품질 기준:

- 관리자 권한이나 화면명은 공개 문서로 확인 가능한 수준에서만 쓴다.
- 실제 계정 설정을 자동 변경한다고 주장하지 않는다.
- 적용 전 반드시 `gaps.md`의 미확인 정책을 운영자가 확인하도록 안내한다.

## 정보 부족 시 동작

URL 접근 실패:

- 해당 URL을 재시도하고, 계속 실패하면 `gaps.md`의 `접근 실패 URL`에 URL, 실패 증상, 확인일을 적는다.
- 실패 URL의 내용을 추정하지 않는다.

동적 로딩/일부 수집:

- 브라우저 본문과 공개 AJAX/정적 리소스에서 합법적으로 접근 가능한 범위만 수집한다.
- 일부만 수집한 경우 `부분 수집`으로 표시한다.

미확인 정책:

- 문서에 없는 정책은 `미확인`으로 표시한다.
- `alf-rules.md`에는 “공개 문서에서 확인되지 않은 정책은 답변하지 말고 상담원에게 연결한다”는 규칙을 둔다.

문서 충돌:

- 충돌하는 문장과 URL을 함께 기록한다.
- 최신성이나 우선순위를 공개 문서로 확인할 수 없으면 사람이 결정하도록 한다.

고위험 주제:

- 환불, 결제 취소, 상품 하자 판정, 개인정보, 법적 권리, 건강/의료 효능은 상담원 확인 또는 보수적 안내를 기본값으로 둔다.
- 상품별 반품 가능 기간이나 배송비가 다르면 고객에게 상품명/주문 상태 확인을 요청한다.
