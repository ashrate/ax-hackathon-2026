# ChannelTalk ALF Knowledge Planner

## 목적

채널톡의 공식 영상 힌트인 “이커머스 고객사의 상담 문제”와 “AI 에이전트”에 맞춰, 공개 사이트/FAQ/정책 문서를 ALF 도입 전 지식·규칙·테스트 세트로 구조화한다. 목표는 조사 보고서가 아니라, 사용자가 반복 실행할 수 있는 ALF 준비 워크플로우를 제공하는 것이다.

## 먼저 읽을 자료

작업을 시작할 때 `references/evidence.md`를 기준 근거 맵으로 삼는다. 사용자가 새 공개 출처를 주면 이 파일의 원칙에 맞춰 출처 강도와 확인 필요 여부를 표시한다.

## 발동 조건

다음 요청에서 사용한다.

- 이커머스 FAQ, 배송/교환/환불 정책, 상품 안내, 쿠폰/프로모션 문서를 ALF용 지식으로 바꿀 때
- 채널톡 트랙의 5문항 답변, README, 검증 기록을 이 문제 기준으로 정리할 때
- 공개 출처가 충분한지, 답변 규칙과 상담원 이관 기준이 안전한지 점검할 때

## 입력

- 공개 URL 또는 사용자가 붙여넣은 FAQ/정책/상품 설명
- 선택 입력: 대상 범위(배송, 교환/환불, 사이즈, 쿠폰, 주문 변경, 상담원 이관 등)
- 선택 입력: 금지 답변, 브랜드 톤, 상담원 확인이 필요한 정책

## 수행 단계

1. 입력 자료와 `references/evidence.md`의 공개 근거를 분리해 기록한다.
2. 문서를 `배송`, `교환/환불`, `주문 변경`, `상품/사이즈`, `쿠폰/프로모션`, `회원/계정`, `상담원 이관` 같은 상담 의도 단위로 나눈다.
3. 각 의도마다 답변에 반드시 필요한 근거 문장, 사용 가능한 답변 규칙, 답변하면 안 되는 조건, 상담원 이관 조건을 만든다.
4. 정책 민감도가 높은 항목은 `high risk`로 표시하고, 사람이 확인해야 할 질문을 붙인다.
5. 근거가 부족하거나 문서끼리 충돌하면 추정하지 않고 `missing` 또는 `conflict`로 표시한다.
6. ALF 테스트 질문을 정상 질문, 애매한 질문, 정책 충돌 질문, 상담원 이관 질문으로 나눠 만든다.
7. 사용자가 제출 문항 작성을 요청하면 같은 문제와 작동 방식이 README와 5문항에 동시에 반영되도록 초안을 낸다.

## 출력 형식

기본 출력은 다음 순서를 따른다.

1. 입력 범위와 공개 출처 표
2. ALF 지식 맵
3. AI 답변 규칙과 금지 답변
4. 상담원 이관 기준
5. 누락·충돌 정보 체크리스트
6. 테스트 질문과 기대 근거
7. 사람이 최종 확인해야 할 항목

## 주의사항

- 채널톡 내부 ALF 설정이나 고객사의 비공개 주문/CRM 데이터를 안다고 쓰지 않는다.
- 환불, 취소, 배송 예정일, 쿠폰 적용처럼 고객 불만으로 이어질 수 있는 항목은 공식 문서 근거 없이는 단정하지 않는다.
- 공개 출처가 약한 주장은 `확인 필요`로 표시한다.
- API 키, 계정 정보, 개인정보는 대화와 로그에 남기지 않는다.
- 로그 제출 조건 때문에 대화 내용을 임의로 편집하거나 요약본만 제출하라고 안내하지 않는다.
