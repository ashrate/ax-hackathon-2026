# 카카오페이증권 제출용 플러그인 (kakaopay-securities-plugin)

> AX 인재전쟁 2026 예선 제출 후보 — 카카오페이증권 트랙

## 1. 플러그인 개요

이 플러그인은 카카오페이증권 또는 투자자가 겪는 실제 문제를 공개 자료로 입증하고, 그 문제를 해결하는 Codex 플러그인 제출물을 설계하도록 돕는 스킬 패키지입니다. 최종 제출 전에는 하나의 문제, 공개 출처, 작동 예시, 검증 결과를 이 문서에 확정해서 반영해야 합니다.

## 2. 문제 정의 기준

문제는 다음 네 가지가 모두 드러나야 합니다.

1. 누가 겪는 문제인지: 카카오페이증권 내부 팀, 개인 투자자, 신규 투자자, 고객지원 담당자 중 하나로 명확히 지정
2. 어떤 상황에서 발생하는지: 투자 정보 탐색, 리스크 이해, 고객 문의, 상품 설명, 공시/뉴스 해석 등 반복 업무 맥락
3. 현재 방식의 한계: 정보가 흩어짐, 금융 용어가 어렵고, 사람이 반복 설명함, 판단 근거가 불명확함 등
4. 플러그인이 바꾸는 점: Codex가 자료 수집, 분류, 초안 작성, 검증 체크를 반복 가능하게 수행

### 공개 근거 기록 형식

공개 출처만 사용합니다. 내부 추정, 출처 없는 숫자, 비공개 대화는 제출 근거로 쓰지 않습니다.

## 3. 설치·실행 방법

1. `submission.zip`을 생성합니다.

```bash
python tools/make_submission.py kakaopay-securities
```

2. 생성된 `dist/kakaopay-securities/submission.zip` 안의 `src/` 폴더가 플러그인 루트입니다.
3. Codex에서 플러그인을 로드한 뒤 아래와 같은 문구로 스킬을 호출합니다.

```text
카카오페이증권 공개 자료로 문제 후보를 찾아줘.
이 투자자 문제를 Codex 플러그인으로 구체화해줘.
카카오페이증권 트랙 5문항 답변을 검토해줘.
```

## 4. 검증 방법

제출 전 아래 항목을 확인합니다.

1. `src/.codex-plugin/plugin.json`이 validator를 통과한다.
2. `src/skills/main/SKILL.md`에 실제 수행 절차와 금지사항이 들어 있다.
3. `README.md`에는 확정된 문제, 공개 출처, 작동 방식, 검증 결과가 들어 있다.
4. `logs/`에는 AI와 나눈 전체 대화 로그 원본이 포함되어 있다.
5. 제3자가 같은 입력으로 스킬을 호출했을 때 문제 정의, 공개 근거 표, 플러그인 작동 흐름을 재현할 수 있다.

## 5. 5문항 답변 작성 기준

### ① 무엇을, 누가, 어떤 상황에서 쓰나요?

만든 플러그인을 한 문장으로 요약합니다. 이어서 카카오페이증권의 누가, 어떤 상황에서, 어디서 막혀 이 플러그인을 쓰게 되는지 구체적으로 적습니다. 카카오페이증권과 관련된 구체적 사용자와 반복 업무 상황을 한 문단으로 정의합니다.

제한: 800자

### ② 왜 이 문제를 선택했나요?

이 문제를 고른 이유와, 카카오페이증권의 직원이나 고객 또는 투자자가 실제로 어디서 왜 막히는지 적습니다. 공개 자료가 입증하는 불편, 시장 변화, 업무 반복성, Codex 플러그인으로 해결 가능한 이유를 연결합니다.

제한: 800자
필수: 출처 URL

### ③ 플러그인은 어떻게 작동하나요?

플러그인이 어떻게 동작하는지, 절차, 지식, 판단 기준을 어떻게 가져가는지 적습니다. 정보가 부족하거나 잘 안 풀리는 상황에서의 동작도 함께 적습니다. 트리거, 입력, Codex의 단계별 작업, 출력물, 사람이 검토하는 지점을 설명합니다.

제한: 1000자
주의: 세부 내용은 이 README에도 적혀 있어야 합니다.

### ④ AI를 어떻게 활용했나요?

AI에게 맡긴 작업과 직접 판단한 부분을 구분해 적습니다. 만들면서 막혔던 지점, 해결 과정, 받아들이지 않은 AI 제안과 그 이유도 적습니다. 문제 조사, 근거 정리, 스킬 설계, 코드/문서 검토, 패키징 검증 과정에서 AI가 수행한 역할을 사실대로 씁니다.

제한: 800자

### ⑤ 어떻게 검증했나요?

입력에서 결과로 이어지는 예시를 하나 이상 적습니다. 정상 상황과 예외 상황을 어떻게 확인했는지, 무엇을 의심했고 무엇이 아직 부족한지, 테스트하며 고친 점을 적습니다. validator, 패키징 명령, 실제 스킬 호출 결과, 공개 출처 재확인 결과를 기준으로 검증합니다.
