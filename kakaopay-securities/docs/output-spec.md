# 카카오페이증권 출력 계약

## 목적

상담사 또는 VOC 담당자가 고객 문의 텍스트와 시간대, 채널, 장애 여부 같은 맥락을 붙여넣으면 플러그인은 감사 문서가 아니라 바로 검토하고 보낼 수 있는 케이스 산출물을 만든다. 모든 실행의 종착점은 `output/case-<번호>/` 폴더이며, 예시 실행은 같은 파일명을 `examples/case-<번호>/`에 둔다.

## 입력 계약

필수 입력:

- 고객 문의 원문

권장 입력:

- 접수일시와 시간대
- 접수 채널: 앱, 고객센터, VOC, 게시판, 기타
- 상품 또는 기능 맥락: 소수점매매, 국내주식, 해외주식, 주문장애, 투자상담 요청 등
- 상담사가 이미 작성한 답변 초안
- 현재 고객이 원하는 행동: 주문 취소, 보상 접수, 일반 설명, 투자 판단 등

입력 금지:

- 계좌번호, 주민등록번호, 전화번호, 주소, 주문번호 전체값 같은 개인정보
- 비공개 내부 장애 로그나 고객 계좌 내역
- API 키, 계정 정보, 사내 접속 토큰

## 출력 폴더

각 실행은 다음 구조를 만든다.

```text
output/case-<번호>/
  response-draft.md
  checklist.md
  compliance-report.md
  escalation.md
  case-log.csv
```

`case-<번호>`는 기존 `output/case-*` 중 가장 큰 번호 다음 번호를 쓴다. 예시 또는 제출 검증 실행은 `examples/case-<번호>/`에 같은 구조로 저장한다.

## `response-draft.md`

고객에게 그대로 보낼 수 있는 답변 문안이다.

필수 스키마:

```markdown
# 고객 답변 초안

안녕하세요, 카카오페이증권입니다.

<고객에게 보낼 본문>

<필수 유의 문구>

## 근거

[^1]: <공식 출처명>, <URL>
```

품질 기준:

- 쉬운 한국어로 작성한다. 전문용어는 한 번 풀어쓴다.
- 종목, 가격, 목표가, 매수·매도 타이밍을 추천하지 않는다.
- 공식 URL 각주를 최소 1개 이상 포함한다.
- 소수점 주문은 체결 구조와 최신 설명서 확인 필요성을 함께 안내한다.
- 장애·보상 문의는 보상 가능 여부를 단정하지 않고 공식 접수·로그·비상주문 시도·신청기한 확인이 필요하다고 쓴다.
- 투자 판단 문의는 추천 대신 목적, 기간, 손실 감내도, 주문 방식 이해 여부를 확인하는 과정으로 전환한다.
- 필수 유의 문구는 케이스에 맞게 포함한다.
  - 투자 판단: "특정 종목의 매수·매도 판단이나 수익을 보장하는 안내는 제공할 수 없습니다."
  - 원금 손실: "주식 투자는 원금 손실 가능성이 있습니다."
  - 보상: "보상 여부는 공식 접수 후 기준 충족 여부를 확인해야 하며, 이 답변만으로 보상을 확정하지 않습니다."
  - 최신성: "거래 조건과 공지는 변경될 수 있어 공식 FAQ·공지의 최신 내용을 함께 확인해 주세요."

정보 부족 시 동작:

- 문의가 모호하면 답변을 단정하지 않고 확인 질문을 먼저 제시한다.
- 공식 근거를 찾지 못한 주제는 "공식 확인 필요"라고 표시하고 고객센터 또는 담당 부서 확인으로 넘긴다.
- 고객 계좌나 주문 내역 확인이 필요한 경우 공개 채널에 세부정보를 쓰지 말고 공식 인증 채널로 안내한다.

## `checklist.md`

상담사가 답변 전 확인해야 할 질문과 다음 행동 단계다.

필수 스키마:

```markdown
# 확인 체크리스트

## 케이스 요약

| 항목 | 내용 |
| --- | --- |

## 고객에게 물어볼 확인 질문

| 질문 | 이유 | 필수 여부 |
| --- | --- | --- |

## 다음 행동 단계

1. <행동>
```

품질 기준:

- 고객의 목적, 투자 기간, 손실 감내도, 주문 방식 이해 여부를 투자 판단이 필요한 케이스에서 반드시 묻는다.
- 장애·보상 케이스는 발생 시각, 접속 채널, 오류 화면, 비상주문 시도 여부, 보상 신청 경로를 확인한다.
- 소수점 주문 케이스는 주문 시간, 주문 유형, 주문 취소 가능 시점, 실시간 체결 기대 여부를 확인한다.
- 개인정보를 평문으로 받지 말고 인증된 고객센터 채널에서만 확인하라고 쓴다.

정보 부족 시 동작:

- 고객 질문만으로 답변 위험이 높으면 `response-draft.md`는 짧은 접수 안내로 만들고, `checklist.md`의 확인 질문을 우선 출력한다.

## `compliance-report.md`

답변 초안의 금지 표현 자체 검사 결과다.

필수 스키마:

```markdown
# 컴플라이언스 자체 점검

## 검사 대상

<고객 문의와 답변 초안 요약>

## 금지 표현 플래그

| 검사 항목 | 판정 | 근거 | 수정 제안 |
| --- | --- | --- | --- |

## 최종 판정

<발송 가능 / 수정 후 발송 / 이관 필요>

## 근거

[^1]: <공식 출처명>, <URL>
```

검사 항목:

- 종목·가격·타이밍 추천성 표현
- 보상 단정 표현
- 원금·수익 보장성 표현
- 불확실한 사항을 확실하다고 오인하게 하는 표현
- 고객 개인정보 또는 비공개 계좌정보 요구 표현

품질 기준:

- 금융위원회, 금융감독원, 금융투자협회 등 공식·자율규제 근거 URL을 인용한다.
- "오를 가능성이 높다", "지금 매도하세요", "보상됩니다", "손해 아닙니다", "원금은 안전합니다" 같은 표현은 플래그한다.
- 수정 제안은 고객이 다음 행동을 알 수 있는 문장으로 바꾼다.

정보 부족 시 동작:

- 규제 판단이 애매한 문구는 "수정 후 발송"으로 두고 더 보수적인 표현을 제안한다.
- 공식 근거가 없으면 "공식 확인 필요"와 함께 발송 보류 또는 이관으로 판정한다.

## `escalation.md`

이관과 라우팅 판단이다.

필수 스키마:

```markdown
# 이관 판단

## 라우팅 결과

| 구분 | 판단 |
| --- | --- |

## 이관 조건

| 조건 | 이관처 | 근거 |
| --- | --- | --- |

## 상담사 처리 문구

<내부 처리자가 다음 단계에서 쓸 수 있는 문구>

## 근거

[^1]: <공식 출처명>, <URL>
```

라우팅 기준:

- 장애·보상 문의: 카카오페이증권 주문장애 기준과 비상주문·증빙·신청기한 절차로 이관한다.
- 앱 접속 지연: 공지 확인 후 장애·보상 기준 충족 여부 확인으로 이관한다.
- 투자자문 요구: 특정 종목 매수·매도 판단을 하지 않고 정식 등록 투자자문업자 안내 또는 객관 정보 제공으로 전환한다.
- 공식 근거 부족: "공식 확인 필요"로 표시하고 담당 부서 또는 고객센터 확인으로 넘긴다.

정보 부족 시 동작:

- 고객이 장애인지 단순 체결 지연인지 구분할 수 없으면 장애 판단을 유보하고 사실관계 확인 질문을 먼저 낸다.
- 투자 판단 요청이지만 고객이 일반 정보만 원한다고 정정하면 객관 정보 안내로 처리한다.

## `case-log.csv`

케이스 원장이다. 각 케이스 폴더에는 현재 케이스까지 반영한 원장 스냅샷을 둔다. 실제 운영에서는 중앙 원장에 같은 행을 누적할 수 있다.

필수 헤더:

```csv
날짜,문의유형,위험플래그,사용근거URL,처리결과
```

품질 기준:

- 날짜는 `YYYY-MM-DD` 형식으로 쓴다.
- `문의유형`은 소수점 주문, 앱 접속 장애/보상, 투자 판단 요청, 기타 중 하나를 포함한다.
- `위험플래그`에는 추천성, 보상 단정, 원금·수익 보장, 공식 확인 필요, 개인정보 주의 중 해당 항목을 적는다.
- `사용근거URL`은 세미콜론으로 구분한 공식 URL 목록만 적는다.
- `처리결과`는 답변 초안 생성, 수정 후 발송, 이관 필요, 공식 확인 필요 중 하나를 포함한다.

정보 부족 시 동작:

- 공식 근거가 없으면 `사용근거URL`에 `공식 확인 필요`라고 쓰고 `처리결과`도 `공식 확인 필요`로 남긴다.
- 고객 원문이 개인정보를 포함하면 원장에 원문을 옮기지 않고 `개인정보 포함 입력 - 원문 미기재`라고 처리 결과에만 기록한다.
