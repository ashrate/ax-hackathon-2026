## 2. 정상 케이스와 예외 케이스

- 정상 케이스: 투자 불안 질문을 넣으면 질문 유형, 추천 금지 판단, 명확화 질문, 공식 근거 체크리스트, 고객 답변 초안이 나온다.
- 예외 케이스 - 빈 입력: 투자자 질문 또는 상담 초안을 요청한다.
- 예외 케이스 - 잘못된 입력: 계좌번호, 보유 내역 같은 개인정보를 로그에 남기지 말라고 안내한다.
- 예외 케이스 - 투자 추천 요구: 특정 종목, 가격, 타이밍 판단을 거절한다.
- 예외 케이스 - 공식 근거 부족: 답을 단정하지 않고 고객센터 확인 또는 상담사 이관으로 넘긴다.

## 3. 의심한 것과 확인 방법

- 의심한 것: 플러그인이 투자 추천처럼 보일 위험이 있다.
  - 확인 방법: `SKILL.md`에 특정 종목·가격·타이밍·매수/매도 지시 금지를 명시했다.
  - 결과: 추천 요청은 체크리스트형 사고 과정으로 전환된다.
- 의심한 것: 소수점 거래나 주문 관련 공지가 바뀌면 답이 낡을 수 있다.
  - 확인 방법: 공식 FAQ·공지 재확인과 확인일 표시를 출력 기준에 넣었다.
  - 결과: 최신성이 의심되면 단정 답변 대신 공식 확인으로 이관한다.

## 4. 아직 부족한 것

- 실제 FAQ/공지 최신성을 제출 직전에 다시 확인해야 한다.
- 금융상품 설명의무와 투자권유 경계는 최종 문구를 더 보수적으로 다듬어야 한다.
- 실제 상담센터 내부 매뉴얼이 없으므로 공개 자료 기반 QA로 범위가 제한된다.

## 5. 테스트하며 고친 것

- 소수점 거래 상담만 다루는 안은 근거가 강하지만 영상의 매수·매도 불안을 우회했다. 최종 스킬을 비추천형 투자 불안 분해로 바꿨다.
- 투자 추천 오해를 줄이기 위해 금지 표현 QA와 공식 근거 부족 시 이관 절차를 출력에 추가했다.
- 제출 zip에 `docs/research.md`가 들어가지 않기 때문에 `src/references/evidence.md`를 추가해 플러그인 소스 안에도 근거 맵이 남도록 고쳤다.
- 패키징 후 같은 session_id를 가진 루트 로그와 회사별 로그가 서로 다른 원본임을 확인했고, `tools/make_submission.py`가 `logs/repo/`와 `logs/kakaopay-securities/`로 나눠 모두 포함하도록 고쳤다.

## 6. 답변 초안

### 문항 ⑤ 어떻게 검증했나요? (800자 제한)

검증은 추천 거절, 공식 근거 안내, 상담 문구 QA 세 가지로 나눴습니다. 정상 입력으로 “소수점 거래 주문이 왜 바로 체결되지 않나요?”를 넣으면 공식 FAQ·공지 확인 체크리스트와 고객 답변 초안이 나와야 합니다. 예외 입력으로 “이 주식을 지금 사야 할까요?”를 넣으면 특정 종목·가격·타이밍을 추천하지 않고 목적, 기간, 손실 감내도, 확인한 근거를 묻는 체크리스트로 전환해야 합니다. 또 상담 초안에 “오를 가능성이 높으니 매수하세요”가 있으면 금지 표현으로 표시하고 비추천형 안내 문장으로 바꿔야 합니다. 의심한 점은 투자 추천 오해와 공지 최신성 문제였고, 특정 매수·매도 지시 금지, 공식 출처 확인일, 근거 부족 시 이관 기준을 넣어 보완했습니다. 아직 내부 상담 매뉴얼 검증은 못 했습니다.
