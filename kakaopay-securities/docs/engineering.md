# 카카오페이증권 — 엔지니어링 기록

> 이 파일은 submission.zip에는 포함되지 않는 작업 기록이며, 문항 ③·④와 README.md 작성의 원천 자료다.

## 1. 플러그인 설계 개요

- 플러그인명: KakaoPay Securities Risk Review Copilot
- 해결 문제: 고객 문의에는 거래 구조 설명, 손실 불안, 보상 기대, 투자 판단 요청이 섞여 있다. 상담사와 VOC 담당자는 이를 빠르게 답해야 하지만, 투자 추천·보상 단정·원금/수익 보장 표현을 피하면서 공식 근거를 붙이기 어렵다.
- 사용자: 카카오페이증권 상담사 또는 VOC 담당자
- 실행 입력: 고객 문의 원문, 접수 시간대, 채널, 상품·기능 맥락, 상담 초안
- 실행 출력: `output/case-<번호>/response-draft.md`, `checklist.md`, `compliance-report.md`, `escalation.md`, `case-log.csv`

## 2. 출력 계약

모든 실행은 감사 메모가 아니라 파일 산출물로 끝난다.

```text
output/case-<번호>/
  response-draft.md
  checklist.md
  compliance-report.md
  escalation.md
  case-log.csv
```

- `response-draft.md`: 상담사 또는 컴플라이언스 담당자가 검토 후 사용하는 답변 초안. 공식 URL 각주와 유의 문구, 사람 검토 필수 문구 포함.
- `checklist.md`: 고객에게 물어볼 확인 질문과 다음 행동.
- `compliance-report.md`: 추천성, 보상 단정, 원금·수익 보장, 불확실한 단정, 개인정보 요구 검사.
- `escalation.md`: 장애·보상, 투자자문, 공식 근거 부족 상황의 라우팅.
- `case-log.csv`: 날짜, 문의유형, 위험플래그, 사용근거URL, 처리결과.

세부 스키마는 `docs/output-spec.md`에 확정했다.

## 3. 작동 절차

1. 다음 케이스 번호와 출력 폴더를 정한다.
2. 문의를 소수점 주문·체결 지연, 앱 접속 장애·보상, 투자 판단 요청, 일반 거래 구조, 상담 초안 QA로 분류한다.
3. 불안을 체결 지연, 손실 가능성, 보상 가능성, 매도·보유 판단, 주문 방식 이해 부족으로 분해한다.
4. `src/references/evidence.md`에서 공식 FAQ·공지·금융위·금감원·금투협 근거를 매칭한다.
5. 확인 가능한 사실과 확인 불가능한 판단을 분리한다.
6. 계약된 5개 파일을 생성한다.
7. 금지 표현 자체 검사를 수행하고, 남은 위험 문장을 수정 제안으로 바꾼다.

## 4. 판단 기준과 폴백

- 특정 종목·가격·목표가·매수/매도 타이밍을 추천하지 않는다.
- 보상 가능 여부를 단정하지 않는다. 주문장애 기준, 비상주문 시도, 증빙, 신청기한 확인으로 넘긴다.
- 원금 손실이 없거나 수익이 보장된다고 쓰지 않는다.
- 공식 출처가 없으면 "공식 확인 필요"로 표시한다.
- 계좌번호, 주문번호 전체값, 주민등록번호, 전화번호 등 개인정보는 산출물에 적지 않는다.
- 문의가 모호하면 답변보다 확인 질문을 우선한다.

## 6. AI 활용 기록

- AI에 맡긴 작업: 자막·FAQ·공지·금융당국 자료 요약, 공식 근거 매칭, 출력 파일 스키마 작성, 3개 대표 문의의 예시 산출물 생성, 금지 표현 자체 검사.
- 직접 판단한 부분: 결과물을 감사 문서가 아니라 상담/VOC 업무 산출물로 정의했다. 각 실행의 끝을 `output/case-<번호>/` 파일 생성으로 고정했고, 예시도 같은 계약 파일로 만들었다.
- 받아들이지 않은 AI 제안: 종목 분석, 매매 타이밍 보조, 보상 가능성 추정 문구는 투자권유·보상 단정 오해가 커서 제외했다.
- 막혔던 지점: 카카오페이 고객센터 FAQ는 일반 fetch에서 본문이 비어 보였다.
- 해결 과정: `curl` 원문에서 FAQ의 Next.js `qnaList` 데이터를 확인하고, 제출 문서에는 로그인 없이 접근 가능한 공식 URL과 확인일을 남겼다.

## 7. 외부 리뷰 피드백 수용과 반영

- 피드백 1: 직접 발송 가능한 문안처럼 읽히는 표현은 금융 영역에서 위험하다. 반영: README, `SKILL.md`, `docs/output-spec.md`, `src/references/evidence.md`, `examples/case-*/response-draft.md`를 "상담사 또는 컴플라이언스 담당자가 검토 후 사용하는 답변 초안"으로 바꿨고, 사람 검토를 마지막 게이트로 명시했다.
- 피드백 2: 플러그인이 답변 생성기처럼 보인다. 반영: 포지셔닝을 "위험 문구 제거 + 공식 근거 정리 + 에스컬레이션 판단 보조"로 바꾸고, 보상 판단·투자자문 결론은 사람이 승인하도록 정리했다.
- 피드백 3: FSS 루트 URL은 curl 기준 불안정하다. 반영: 전수 URL 점검에서 해당 URL만 `000`을 확인했고, curl `200`이 확인된 금융투자협회 제도권 금융투자회사 조회 페이지와 기존 금융위 구체 페이지로 교체했다.
- 피드백 4: 의사결정 감소 서사가 약하다. 반영: README 전면에 입력 예시, 실행 결과, 실패 시 동작, 사람이 최종 판단하는 지점을 배치하고 상담사의 반복 확인 작업을 표로 정리했다.

## 8. 문항 ③ 초안

이 플러그인은 상담사 또는 VOC 담당자가 고객 문의를 붙여넣으면 `output/case-<번호>/` 아래에 5개 파일을 만듭니다. 먼저 문의를 소수점 주문·체결 지연, 앱 접속 장애·보상, 투자 판단 요청 등으로 분류하고 고객의 불안을 체결 지연, 손실 가능성, 보상 기대, 매도·보유 판단, 주문 방식 이해 부족으로 나눕니다. 이후 `references/evidence.md`의 카카오페이증권 FAQ·공지, 주문장애 기준, 금융위·금투협 자료를 매칭합니다. 출력은 상담사 또는 컴플라이언스 담당자가 검토 후 사용하는 `response-draft.md`, 확인 질문과 행동 단계 `checklist.md`, 추천성·보상 단정·원금/수익 보장 표현을 검사하는 `compliance-report.md`, 장애·보상 또는 투자자문 이관 기준 `escalation.md`, 누적 원장 `case-log.csv`입니다. 문의가 모호하면 확인 질문을 먼저 만들고, 공식 근거가 없으면 "공식 확인 필요"로 표시합니다. 사람 검토 전에는 고객 발송, 보상 확정, 투자자문 결론으로 쓰지 않습니다.

## 9. 문항 ④ 초안

AI에는 공식 영상 자막, 카카오페이증권 FAQ·공지, 금융위·금투협 자료를 근거 맵으로 정리하고, 위험 문구 제거와 공식 근거 정리 중심의 출력 파일 스키마를 설계하게 했습니다. 또 소수점 주문 지연, 앱 접속 장애·보상, 매도 판단 요청 3개 문의를 실제로 실행해 검토용 답변 초안, 체크리스트, 컴플라이언스 점검, 이관 판단, 케이스 원장을 만들게 했습니다. 직접 판단한 부분은 결과물의 성격입니다. 외부 리뷰를 반영해 직접 발송 가능한 문안처럼 읽히는 표현을 사람 검토 후 사용하는 초안으로 바꿨고, FSS 루트 URL은 curl 실패로 교체했습니다. 받아들이지 않은 제안은 종목 분석, 매매 타이밍 추천, 보상 가능성 추정입니다. 규제 오해가 커서 추천·보상 단정을 금지하고 공식 근거와 확인 절차 중심으로 제한했습니다.
