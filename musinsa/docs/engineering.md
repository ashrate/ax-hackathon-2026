# 무신사 — 엔지니어링 기록

> 이 파일은 submission.zip에는 포함되지 않는 작업 기록이며, 문항 ③·④와 README.md 작성의 원천 자료다.

## 1. 플러그인 설계 개요

- 플러그인명: MUSINSA Trend & Brand Triage
- 해결 문제: 무신사 MD, 브랜드 스카우트, 카테고리/뷰티, 마케팅·전략 담당자가 매주 공개 자료에 흩어진 브랜드/협업/입점 후보 신호를 `신규 검토`, `협업/콘텐츠 확장`, `모니터링`, `제외`로 triage하기 어렵다.
- 스킬 구성: `src/skills/main/SKILL.md` 하나로 실행한다. 근거 맵은 `src/references/evidence.md`에 둔다.
- 핵심 입력: 카테고리, 고객군, 트렌드 질문, seed brand, 공개 URL 목록.
- 핵심 출력: 트렌드 신호 ledger, 브랜드 후보 scorecard, supporting/opposing evidence, 출처 강도, 4분류 triage 상태, 다음 액션, 모니터링 쿼리.

## 2. 작동 절차

1. 입력 질문을 브랜드 후보 triage, 트렌드 감지, 카테고리 비교, 후보 모니터링 중 하나로 분류한다.
2. 공개 출처를 무신사 내부 공개 신호, 외부 시장 신호, 고객 반응, 보조 기사/검색으로 구분한다.
3. 후보 브랜드나 트렌드별로 지지 근거와 반대 근거를 함께 기록한다.
4. 출처 강도, 반복 관찰 가능성, 고객 반응 신호, 무신사 카테고리 적합성, 다음 액션을 점수화한다.
5. 접근 제한, 로그인 필요, 출처 불명, 단일 바이럴 노이즈는 약한 근거로 표시한다.
6. 결과는 입점 추천이 아니라 “검토할 후보 가설”과 4분류 triage 상태로 표현한다.
7. 5문항 또는 README 작성 요청이 있으면 같은 문제 정의와 작동 방식이 유지되도록 초안을 낸다.

## 3. 판단 기준

- 공식 영상 자막의 “고객들이 좋아할 브랜드”와 “관심 트렌드를 놓치지 않는 AI” 힌트를 최우선으로 둔다.
- 출처가 강한 반복 신호를 단일 바이럴보다 높게 본다.
- 무신사 랭킹처럼 이미 입점한 신호와 외부에서 떠오르는 신호를 구분한다.
- 후보 브랜드는 정답이 아니라 검토 가설이다.
- 무신사 내부 매출, 검색, CRM, 파트너 데이터 접근을 전제하지 않는다.

## 4. 정보 부족·실패 상황 폴백

- 공개 URL이 없을 때: 사용자의 카테고리와 seed brand만으로 필요한 리서치 쿼리부터 만든다.
- 출처가 로그인 전용일 때: unavailable로 기록하고 핵심 근거에서 제외한다.
- 신호가 한 출처에만 있을 때: weak로 표시하고 반대 근거 탐색을 요구한다.
- 브랜드명 중복이나 동명이 있을 때: 공식 사이트, 판매 채널, 카테고리로 구분한다.
- 트렌드 여부가 모호할 때: 확정하지 않고 “모니터링 후보”로 낮춘다.

## 6. 외부 리뷰 피드백 수용과 반영

- 지적받은 점: 후보 8개 중 여러 브랜드가 이미 무신사/29CM 노출·협업 이력이 있어 "신규 브랜드 발굴"로 보이면 부정확하다.
- 수용한 판단: 이 지적은 맞다. 플러그인이 줄이는 일은 순수 신규 발굴이 아니라 매주 공개 신호를 훑어 후보를 `신규 검토`, `협업/콘텐츠 확장`, `모니터링`, `제외`로 분류하는 반복 triage다.
- 반영한 변경: README 전면부를 입력 예시, 실행 결과, 실패 시 동작, 사람 최종 판단 지점으로 재배치했고, 예시 실행은 8개 후보를 `협업/콘텐츠 확장` 3개와 `모니터링` 5개로 설명했다.
- 지적받은 점: 결과 다수가 manual-check라 플러그인이 의사결정을 줄였다는 느낌이 약하다.
- 수용한 판단: 동적 화면 확인은 사람이 해야 하지만, 후보별 URL 수집, 출처 강도, 반대 근거, 이미 노출된 브랜드 분기, 다음 액션 정리는 플러그인이 대신한다.
- 반영한 변경: README에 수동 작업 대비 플러그인이 대신 산출하는 항목 표를 추가했다.
- 지적받은 점: Apparelnews/Fashionbiz 일부 링크가 curl 기준 403이다.
- 수용한 판단: 403 URL은 제출 근거로 불안정하다.
- 반영한 변경: Fashionbiz/Apparelnews 링크는 대상 문서와 examples에서 제거했다. THE BARNNET은 29CM/KTNews, SIYAZU는 공식 스토어/29CM, LE17SEPTEMBRE는 공식 사이트/29CM, TREEMINGBIRD는 29CM/무신사/Fashion Insight로 교체했다. 교체 불가한 컬렉션·잡화 확장 주장은 모니터링 가설로 낮췄다.

## 7. AI 활용 기록

- AI에 맡긴 작업: 자막·뉴스룸·랭킹·리뷰 기반 후보 정리, 영상 의도와 후보 문제의 정합성 비교, 스킬 출력 구조 설계.
- 직접 판단한 부분: 입점 후 상품 운영 QA보다 브랜드/트렌드 공개 신호 triage가 출제 영상의 직접 질문에 더 맞는다고 결정했다.
- 받아들이지 않은 AI 제안: 파트너 상품·가격·배송 설정 사전 QA는 공개 가이드가 강하지만, 영상의 “고객이 좋아할 브랜드 발굴” 질문보다 후단 운영 문제라 최종 문제에서 제외했다.
- 이번 감사에서 AI에 맡긴 작업: 공식 채용공고, 뉴스룸, 회사 사업 페이지, 파트너 문서, 랭킹/추천 페이지, AI 트렌드 큐레이션 관련 공개 자료를 탐색하고 출처 강도·접근성·반증 가능성을 표로 정리하게 했다.
- 이번 감사에서 직접 판단한 부분: 검색 결과 문구만 확보된 채용공고는 핵심 근거에서 제외했고, 동적 페이지인 랭킹·추천은 문항② 최종 URL에서 제외했다. `AI 트렌드 큐레이션`은 문제를 약화시키는 반증으로 인정하되, 플러그인의 목적을 MD용 후보 가설 감사와 다음 액션 정리로 제한해 차별화했다.
- 이번 감사에서 받아들이지 않은 AI 제안: 랭킹/추천 페이지를 문항② 핵심 URL로 넣는 안은 HTTP 200이어도 본문 텍스트 추출이 제한되어 제외했다. 대신 본문 추출이 안정적인 공식 채용공고와 뉴스룸 URL을 최종 후보로 올렸다.
- 이번 출력 계약 전환에서 AI에 맡긴 작업: 산출물 폴더 구조, 파일별 스키마, 실제 여성 컨템포러리 후보 브랜드의 공개 출처 수집, 예시 스코어카드 초안 생성을 맡겼다.
- 이번 출력 계약 전환에서 직접 판단한 부분: `여성 컨템포러리`를 예시 카테고리로 정했고, 후보가 이미 플랫폼에 노출된 경우 `신규 검토`가 아니라 `협업/콘텐츠 확장`으로 낮췄다. 동적 페이지에서 자동 추출한 숫자는 쓰지 않고 수동 확인 액션으로 남겼다.
- 이번 출력 계약 전환에서 받아들이지 않은 AI 제안: 공개 기사 숫자만으로 트렌드 확정 또는 입점 추천을 쓰는 안은 제외했다. 출처 강도 3 이하 단독 근거는 `가설` 또는 `모니터링`으로 표시했다.
- 이번 외부 리뷰 반영에서 AI에 맡긴 작업: 문서와 examples 전체의 URL을 추출하고 `curl -L -o /dev/null -s -w "%{http_code}"` 기준으로 상태를 점검하게 했다. 403 URL의 대체 공식/플랫폼 출처 후보를 찾게 했다.
- 이번 외부 리뷰 반영에서 직접 판단한 부분: 문제를 정직하게 `triage 자동화`로 재정의했고, TREEMINGBIRD는 보조 기사 반응이 있어도 공식 플랫폼 재확인 전까지 `모니터링`으로 낮췄다.
- 이번 외부 리뷰 반영에서 받아들이지 않은 AI 제안: 403 기사 링크를 "브라우저에서는 열릴 수 있음"으로 유지하는 안은 제출 안정성이 낮아 제외했다.

## 8. 답변 초안

### 문항 ③ 플러그인은 어떻게 작동하나요? (1000자 제한)

이 플러그인은 카테고리, 고객군, 트렌드 질문, seed brand 또는 공개 URL을 입력받아 무신사 MD와 브랜드 스카우트가 검토할 후보 가설을 만듭니다. 먼저 출력 폴더를 정하고 공개 뉴스룸, 브랜드 공식 사이트, 29CM/무신사 공개 페이지, 기사·커뮤니티를 출처 강도와 curl 접근 상태로 나눕니다. 후보별 supporting evidence와 opposing evidence를 함께 기록한 뒤 `신규 검토`, `협업/콘텐츠 확장`, `모니터링`, `제외`로 triage합니다. 이미 무신사/29CM 노출이 강하면 신규 발굴로 쓰지 않고 확장 후보로 분기합니다. 로그인 전용 자료, 403 URL, 동적 화면, 단일 보조 기사는 weak 또는 manual-check-required로 낮추고 핵심 근거로 쓰지 않습니다. 결과는 `signals-log.csv`, 브랜드별 scorecard, scout-report, watchlist로 냅니다.

### 문항 ④ AI를 어떻게 활용했나요? (800자 제한)

AI에는 유튜브 자막에서 출제자의 문제 힌트를 뽑고, 무신사 뉴스룸·랭킹·브랜드 공식 페이지·공개 기사 URL을 후보 문제와 연결하는 작업을 맡겼습니다. 또 examples의 URL을 curl 기준으로 점검하고 대체 출처 후보를 찾게 했습니다. 직접 판단한 부분은 문제 재정의입니다. 외부 리뷰처럼 기존 예시는 신규 브랜드 발굴보다 이미 알려진 브랜드와 약한 후보를 분류한 결과에 가까웠으므로, 문제를 주간 triage 자동화로 바꿨습니다. 받아들이지 않은 제안은 403 기사 링크를 그대로 두는 안과, 기사 수치만으로 협업 후보를 확정하는 안입니다. 제출 안정성과 근거 강도가 약해 공식/플랫폼 출처로 교체하거나 모니터링으로 낮췄습니다.
