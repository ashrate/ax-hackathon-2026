# Musinsa Trend & Brand Triage Output Spec

> 작성일: 2026-07-08
> 목적: 플러그인이 감사 문서가 아니라 MD/브랜드 스카우트가 바로 들고 갈 수 있는 4분류 triage 산출물을 만들도록 출력 계약을 고정한다.

## 0. 4분류 triage 계약

모든 실행은 후보를 `신규 검토`, `협업/콘텐츠 확장`, `모니터링`, `제외` 중 하나로 분류해야 한다. 이 상태값은 `scout-report.md`, 각 `scorecards/*.md`, `watchlist.md`에 반복해서 나타나는 1급 출력이다.

| 상태 | 의미 | 사용 기준 |
| --- | --- | --- |
| `신규 검토` | 아직 무신사/29CM 노출이 강하지 않고 다음 MD 검토 액션이 명확한 후보 | 4등급 이상 근거와 공개 고객 반응 신호가 함께 있다. |
| `협업/콘텐츠 확장` | 이미 무신사/29CM에 노출·입점·협업 이력이 있어 신규 발굴보다 확장 검토가 맞는 후보 | 단독 상품, 라이브, 콘텐츠, 글로벌, 계절 기획전으로 분기한다. |
| `모니터링` | 방향성은 맞지만 공개 고객 반응, 최근성, 공식 근거 중 하나가 부족한 후보 | 다음 확인 날짜와 필요한 공개 근거를 남긴다. |
| `제외` | 접근 제한, 출처 약함, 동명 혼선, 카테고리 부적합, 운영 리스크로 이번 주 검토에서 빼는 후보 | 근거와 제외 이유를 남기되 추천하지 않는다. |

## 1. 출력 경로 계약

사용자가 카테고리, 고객군, 트렌드 질문, seed brand, 공개 URL 중 하나 이상을 주면 플러그인은 다음 경로를 만든다.

```text
output/<category-slug>-<YYYY-MM-DD>/
```

예시는 제출물 안에서 다음 경로를 쓴다.

```text
examples/<category-slug>-<YYYY-MM-DD>/
```

`category-slug`는 사용자가 준 카테고리를 사람이 읽을 수 있게 보존하되 공백은 `-`로 바꾼다. 예: `여성 컨템포러리` -> `여성-컨템포러리`.

필수 파일 구조:

```text
output/<category-slug>-<YYYY-MM-DD>/
  scout-report.md
  signals-log.csv
  watchlist.md
  scorecards/
    <브랜드>.md
```

## 2. scout-report.md

`scout-report.md`는 주간 스카우트 회의에서 바로 읽는 1차 리포트다.

필수 섹션:

1. 실행 요약
2. 입력 질문과 범위
3. 후보 브랜드 스코어카드 요약표
4. 이번 주 트렌드 신호 3~5개
5. 반대 근거와 검증 필요 지점
6. 추천 다음 액션
7. 사용한 공개 출처 목록

후보 브랜드 요약표 필수 컬럼:

| 컬럼 | 내용 |
| --- | --- |
| 브랜드 | 실존 브랜드명. 동명이 있으면 공식 사이트 또는 플랫폼 URL로 식별한다. |
| 핵심 신호 | 왜 이번 실행에서 후보가 되었는지 1~2문장으로 적는다. |
| 출처 URL·확인일 | 핵심 신호를 뒷받침하는 공개 URL과 확인일을 함께 적는다. |
| 고객 반응 신호 | 거래액, 주문 수량, 방문, 리뷰, 플랫폼 큐레이션, 커뮤니티 반응 등 공개 출처로 확인된 반응만 쓴다. 없으면 `확인 필요`라고 쓴다. |
| 반대 근거 | 이미 충분히 노출됨, 단일 출처, 최근 신호 부족, 카테고리 부적합, 운영 리스크 등 검토를 늦출 이유를 적는다. |
| triage 상태 | `신규 검토`, `협업/콘텐츠 확장`, `모니터링`, `제외` 중 하나와 이유. |
| 추천 다음 액션 | MD/브랜드 스카우트가 다음에 할 구체 액션. 예: 화면 캡처 확인, 브랜드 미팅, 재고/가격대 확인, 랭킹 재조회. |

품질 기준:

- 후보 브랜드는 5~10개를 기본으로 한다.
- 후보마다 supporting evidence와 opposing evidence가 모두 있어야 한다.
- 공개 URL, 확인일, 출처 강도를 함께 남긴다.
- 유행을 단정하지 않고 `가설`, `관찰`, `검토 후보`로 표현한다.
- 내부 매출, 내부 검색량, CRM, 파트너센터 데이터가 있다고 쓰지 않는다.

## 3. scorecards/<브랜드>.md

브랜드별 상세 카드는 입점 검토 회의에 그대로 가져갈 수 있는 1페이지 형식이다.

필수 섹션:

```markdown
# <브랜드> Scorecard

- 카테고리:
- 작성일:
- 상태:
- 한 줄 판단:

## 개요

## 신호와 근거

| 신호 | 출처 URL | 출처 강도 | 확인일 | 해석 |
| --- | --- | --- | --- | --- |

## 고객 반응 신호

## 리스크·반대 근거

## 입점 적합성

## 다음 액션과 담당 제안
```

상태 값:

- `신규 검토`: 4등급 이상 출처가 있고, 고객 반응 또는 플랫폼 반응이 공개 자료로 확인되며, 아직 무신사/29CM 노출이 강하지 않고 다음 액션이 명확하다.
- `협업/콘텐츠 확장`: 이미 무신사/29CM에 노출되어 신규 입점보다 단독 상품, 라이브, 콘텐츠, 글로벌 확장 검토가 더 적합하다.
- `모니터링`: 신호는 있으나 출처 수, 최근성, 고객 반응 중 하나가 부족하다.
- `제외`: 출처 접근 제한, 동명 혼선, 카테고리 부적합, 반대 근거가 큰 경우.

품질 기준:

- 상세 카드 하나는 1페이지 안에서 회의용으로 읽히도록 쓴다.
- 출처 강도 3 이하의 단독 근거만 있으면 상태를 `모니터링` 이하로 둔다.
- 출처 강도 3 이하 단독 근거는 반드시 `가설`이라고 표시한다.
- 4등급 이상 근거라도 동적 페이지에서 본문 추출이 제한되면 사용자가 화면으로 재확인할 액션을 남긴다.

## 4. signals-log.csv

`signals-log.csv`는 반복 실행 때 누적되는 수집 원장이다. 같은 카테고리를 다시 실행할 때 기존 CSV 뒤에 새 행을 추가하는 설계로 둔다.

헤더는 정확히 다음 순서를 지킨다.

```csv
date,signal,source_url,source_strength,related_brand,note
```

컬럼 규칙:

| 컬럼 | 규칙 |
| --- | --- |
| date | 신호를 확인한 날짜. `YYYY-MM-DD` 형식. |
| signal | 관찰한 신호를 한 문장으로 쓴다. |
| source_url | 로그인 없이 접근 가능한 공개 URL. 접근 제한이면 URL을 남기고 note에 `unavailable` 표시. |
| source_strength | `5:official-video`, `4:official/company/platform`, `3:press/review/community`, `2:search-snippet`, `1:ai-summary` 중 하나. |
| related_brand | 관련 브랜드명. 트렌드 공통 신호면 `category`라고 쓴다. |
| note | 해석, 반대 근거, 다음 확인 지점을 짧게 쓴다. |

품질 기준:

- `source_strength`가 3 이하인 행은 후보 확정 근거로 단독 사용하지 않는다.
- 한 브랜드의 핵심 판단에는 가능하면 4등급 이상 근거 1개 이상을 포함한다.
- 검색 스니펫은 후보 탐색에는 사용할 수 있으나 `scout-report.md`의 핵심 근거가 되면 안 된다.

## 5. watchlist.md

`watchlist.md`는 `모니터링`/`제외` 후보와 다음 확인 시점을 관리한다.

필수 섹션:

1. 제외 후보
2. 모니터링 후보
3. 다음 확인 일정
4. 추가로 필요한 입력

필수 컬럼:

| 컬럼 | 내용 |
| --- | --- |
| 후보 | 브랜드명 또는 트렌드명 |
| 현재 판단 | `모니터링`, `제외`, `unavailable`, `가설` 중 하나 |
| 이유 | 왜 본 리포트의 핵심 후보로 올리지 않았는지 |
| 필요한 추가 근거 | 다음에 찾아야 할 공개 자료 |
| 다음 확인 시점 | 날짜 또는 실행 조건 |

정보 부족 시 동작:

- 신호가 부족한 후보는 `scout-report.md`에서 무리하게 추천하지 않고 `watchlist.md`로 내린다.
- 출처 강도 3 이하 단독 근거는 `가설` 표시 후 추가 공식/플랫폼 근거를 요구한다.
- 로그인 전용, 동적 렌더링, 캡처가 필요한 페이지는 `unavailable` 또는 `manual-check-required`로 적고 핵심 근거에서 제외한다.
- 브랜드 공식 사이트와 플랫폼 페이지의 정보가 충돌하면 더 최근 확인일의 공식 페이지를 우선하되, 충돌 내용을 반대 근거에 남긴다.

## 6. 출처 강도와 판단 기준

| 강도 | 라벨 | 예시 | 사용 규칙 |
| --- | --- | --- | --- |
| 5 | official-video | 출제 영상 자막 | 문제 정의와 회사 힌트의 최상위 근거. |
| 4 | official/company/platform | 무신사 뉴스룸, 회사/브랜드 공식 사이트, 29CM/무신사 공개 브랜드 페이지 | 핵심 판단 근거로 사용 가능. |
| 3 | press/review/community | 언론 기사, 앱 리뷰, 공개 커뮤니티 | 보조 근거. 단독이면 `가설`. |
| 2 | search-snippet | 검색 결과 스니펫 | 후보 탐색용. 핵심 근거 금지. |
| 1 | ai-summary | AI 요약, NotebookLM 구조화 결과 | 원문 역추적 전까지 근거로 쓰지 않음. |

최종 판단은 다음 기준으로 정한다.

- `신규 검토`: 공식/플랫폼 근거와 고객 반응 신호가 모두 있고, 아직 무신사/29CM 노출이 강하지 않으며 반대 근거가 실행을 막지 않는다.
- `협업/콘텐츠 확장`: 브랜드가 이미 플랫폼에 있거나 노출이 강해 신규 입점보다 콘텐츠, 단독 상품, 라이브, 글로벌 확장이 더 적합하다.
- `모니터링`: 방향성은 맞지만 공개 고객 반응, 최근성, 공식 근거 중 하나가 부족하다.
- `제외`: 공개 근거가 약하거나 접근 제한이 크거나 카테고리 적합성이 낮다.
