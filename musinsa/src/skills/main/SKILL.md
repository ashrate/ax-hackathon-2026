# MUSINSA Trend & Brand Scout

## 목적

무신사 공식 영상 힌트인 “고객들이 좋아할 브랜드를 어떻게 찾을까”와 “AI로 관심 트렌드를 어떻게 놓치지 않을까”에 맞춰, 공개 자료에서 브랜드 후보와 트렌드 신호를 반복 발굴한다. 목표는 정답 예측이 아니라, 출처와 반대 근거가 붙은 검토 가능한 후보 스코어카드를 만드는 것이다.

## 먼저 읽을 자료

작업을 시작할 때 `references/evidence.md`를 기준 근거 맵으로 삼는다. 사용자가 새 공개 출처를 주면 출처 강도, 확인일, 무신사 내부 신호인지 외부 신호인지 구분한다.

## 발동 조건

다음 요청에서 사용한다.

- 무신사 MD, 브랜드 스카우트, 카테고리/뷰티, 마케팅·전략 담당자의 트렌드 리뷰를 준비할 때
- 공개 랭킹, 뉴스룸, 앱 리뷰, 커뮤니티, 검색 결과를 브랜드 후보 스코어카드로 정리할 때
- 5문항 답변, README, 검증 기록을 이 문제 기준으로 정리할 때

## 입력

- 카테고리, 고객군, 트렌드 질문, seed brand 중 하나 이상
- 선택 입력: 공개 URL 목록, 무신사 랭킹/아카이브 링크, 뉴스룸 기사, 앱 리뷰, 커뮤니티 게시물, 검색 결과
- 선택 입력: 평가 관점(입점 가능성, 협업 가능성, 뷰티 확장성, 고객 반응, 경쟁/리스크 등)

## 수행 단계

1. 입력 질문을 브랜드 발굴, 트렌드 감지, 카테고리 비교, 후보 모니터링 중 하나로 분류한다.
2. 공개 출처를 `무신사 내부 공개 신호`, `외부 시장 신호`, `고객 반응`, `보조 기사/검색`으로 구분한다.
3. 각 후보 브랜드 또는 트렌드에 대해 supporting evidence와 opposing evidence를 함께 남긴다.
4. 후보마다 출처 강도, 반복 관찰 가능성, 고객 반응 신호, 무신사 카테고리 적합성, 다음 액션을 점수화한다.
5. 접근 제한, 로그인 필요, 출처 불명, 과도한 바이럴 노이즈는 `weak` 또는 `unavailable`로 표시한다.
6. 최종 결과는 추천이 아니라 “검토할 후보 가설”로 표현한다.
7. 사용자가 제출 문항 작성을 요청하면 같은 문제와 작동 방식이 README와 5문항에 동시에 반영되도록 초안을 낸다.

## 출력 형식

기본 출력은 다음 순서를 따른다.

1. 리서치 질문과 입력 출처
2. 트렌드 신호 ledger
3. 브랜드 후보 scorecard
4. supporting/opposing evidence
5. 확인일과 출처 강도
6. 다음 액션과 모니터링 쿼리
7. 사람 검토가 필요한 불확실성

## 주의사항

- 무신사 내부 매출, 검색, CRM, 파트너 데이터를 안다고 쓰지 않는다.
- 후보 브랜드를 객관적 정답이나 입점 추천으로 단정하지 않는다.
- 로그인·비공개 SNS 자료는 핵심 근거로 쓰지 않는다.
- 단일 바이럴 자료만으로 트렌드라고 확정하지 않는다.
- API 키, 계정 정보, 개인정보는 대화와 로그에 남기지 않는다.
- 로그 제출 조건 때문에 대화 내용을 임의로 편집하거나 요약본만 제출하라고 안내하지 않는다.
